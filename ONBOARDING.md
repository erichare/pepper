## 1 — Codebase Purpose

pepper is a CLI tool (`pepper archive <cmd>`, defined in `pepper.archive.cli`) that builds a complete archive of one Reddit account's posts and comments — including deleted/removed content — by merging four sources (GDPR export, Arctic Shift, PullPush, Reddit API via PRAW) into a local SQLite database. Downstream stages compute analysis, an LLM persona dossier, and Markdown/HTML reports that feed the pepper bot.

## 3 — Execution Lifecycle

**Representative execution:** `pepper archive backfill` (full history from public archives)

1. **Entry:** `cli.backfill()` builds `Settings.load()`, opens the DB via `storage.connect()` + `apply_migrations()`, and calls `pipeline.backfill()`.
2. **Source selection:** `pipeline._source_order()` constructs `ArcticShiftSource` and `PullPushSource`, each with its own `RateLimiter`; `"auto"` tries Arctic Shift first and falls back to PullPush when a source raises `SourceError`.
3. **Stream walk:** `pipeline._run_archive()` iterates both streams (submissions, comments). For each it reads `Repo.get_watermark()`: an empty watermark means one full pass; otherwise a newer-than-`newest` pass plus an older-than-`oldest` backfill pass.
4. **Pagination:** `ArchiveSource.iter_author_items()` pages newest→oldest through `request_with_retry()`, dedupes by fullname, and yields each row through `map_reddit_obj()` as a `NormalizedItem`.
5. **Persist:** batches of 500 go through `pipeline._flush()` → `Repo.upsert_items()`, which records one row per item+source via `Repo.record_observation()`, then `Repo.recompute_canonical()` calls `merge_item()` to rebuild the canonical `items` row from *all* observations; `Repo.update_watermark()` advances the resume cursor as each batch lands.
6. **Response:** `_run_archive()` closes out the run row via `Repo.finish_run()` and returns per-stream counts, which the CLI prints as JSON.

## 4 — Domain Vocabulary

| Term | Definition |
|---|---|
| **Observation** | One source's view of one item, upserted latest-wins per (item, source) into `source_observations` by `Repo.record_observation()`. |
| **Canonical item** | The single merged row in `items`, recomputed from all observations by `merge_item()`. Never written directly by sources. |
| **Content precedence** | Order for choosing body/title/structural fields: `CONTENT_PRECEDENCE` = gdpr > arcticshift > pullpush > praw (`models.enums`). |
| **Live source** | A source whose scores reflect current Reddit state (`LIVE_SOURCES`); volatile metrics take the freshest live value via `merge._pick_live_metric()`. |
| **Watermark** | Per (source, stream) newest/oldest `created_utc` cursor (`Watermark`, `source_watermarks` table) that makes backfills incremental and resumable. |
| **Fullname** | Reddit's prefixed id (`t1_`/`t3_` + base36), the primary key everywhere; built by `make_fullname()` in `models.ids`. |
| **Context item** | A parent post/comment fetched for the subject's comments (`ContextItem`, `context/fetch.fetch_context()`); stored apart from the subject's own corpus. |
| **Dossier** | The LLM persona profile generated by `llm.dossier.generate_dossier()` (cost-gated, Anthropic Batch API). |

## 5 — Common Change Recipes

### Recipe A — Add a new public archive source

1. Subclass `ArchiveSource` in `sources/`, implementing `_endpoint()` and `_params()` — model on `ArcticShiftSource`.
2. Add a `Source` flag (next power of two — values are frozen) and its `Source.from_name()` mapping in `models/enums.py`; place the name in `CONTENT_PRECEDENCE` and, if scores are current, `LIVE_SOURCES`.
3. Export it from `sources/__init__.py` (`__all__`).
4. Add a rate-limit setting (e.g. `rl_<name>_per_sec`) to `Settings` in `config.py`.
5. Wire it into `pipeline._source_order()`.

### Recipe B — Add a new pipeline stage / CLI command

1. Add a stage function in `pipeline.py`; bracket DB work with `Repo.start_run()` / `Repo.finish_run()` — model on `pipeline.enrich()`.
2. Add an `@archive_app.command()` wrapper in `cli.py` using the `_settings()` / `_open()` / `_guard()` helpers — model on `cli.enrich()`.
3. Sequence it in `pipeline.run_all()` in dependency order.
4. Add a `justfile` recipe wrapping `uv run pepper archive <cmd>`.

### Recipe C — Add a field to the canonical item

1. Add the field to `NormalizedItem` and `CanonicalItem` in `models/item.py`.
2. Populate it in the normalize mappers: `map_reddit_obj()` and/or the `map_gdpr_*_row()` functions.
3. Add pick logic in `merge.merge_item()` (content precedence vs. freshest-live — see existing picks).
4. Add the column in a new numbered SQL file under `storage/migrations/` (discovered by `db._migration_files()`), and bind it in `Repo._upsert_canonical()`.

## 7 — Known Gotchas

### `merge_item` must stay pure

**Rule:** `merge.merge_item()` must remain a pure function of its `NormalizedItem` observations — no I/O, clock reads, or randomness.
**Why:** idempotency rests on it: `Repo.recompute_canonical()` rebuilds the canonical row from all stored observations, and identical observations must yield a byte-identical row.
**What breaks:** canonical rows churn on every re-run — unstable exports and diffs that hide real data changes.

### `Source` bitmask values are frozen

**Rule:** never renumber the `Source` IntFlag values in `models/enums.py` (GDPR=1, PRAW=2, ARCTICSHIFT=4, PULLPUSH=8); new sources take the next power of two.
**Why:** the values are persisted in `items.sources_bitmask` in every existing database.
**What breaks:** renumbering silently corrupts provenance in already-built archives — old rows decode to the wrong source set with no error.

### The GDPR import is gated by a content-hash marker

**Rule:** `pipeline.import_gdpr()` skips the import when `data/raw/gdpr/.imported_hash` matches `GdprSource.content_hash()`. After changing GDPR mapping code (`normalize/gdpr_map.py`), re-run with `--force` (or delete the marker).
**Why:** the marker only tracks input-file changes, not mapper-code changes.
**What breaks:** an edited mapper silently never runs — the database keeps the old normalization and the fix appears to have no effect.
